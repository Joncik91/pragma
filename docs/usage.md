# Pragma usage

This page walks a first-time user through both Pragma flows — adopting Pragma
on an existing repo (brownfield) and starting a new project from empty
(greenfield). Read this before your first `pragma init`.

> **New to Pragma?** Read [`concepts.md`](concepts.md) first for the
> mental model — what the manifest is for, why the gate exists, what
> the PIL is. This page assumes those concepts and shows you how to
> drive the tool. For every flag and field, see
> [`reference.md`](reference.md).

## Install

Install as a standalone CLI with pipx:

```shell
pipx install pragma
```

Or inside a project virtualenv with the dev extra:

```shell
python -m venv .venv
source .venv/bin/activate
pip install "pragma[dev]"
```

Verify:

```shell
pragma --help
```

## Brownfield quickstart

Use brownfield mode when you already have a repo and want to bolt Pragma onto
it.

### 1. Initialise

```shell
cd your-project/
pragma init --brownfield --name your-project
```

This scaffolds four files:

- `pragma.yaml` — the Logic Manifest. Starts empty.
- `pragma.lock.json` — hash of `pragma.yaml`. Regenerated by `pragma freeze`.
- `.pre-commit-config.yaml` — wires `pragma verify all`, `pragma verify
  commits`, and `pragma verify message` into `pre-commit`, `pre-push`, and
  `commit-msg` stages.
- `PRAGMA.md` — a README-style primer for humans and Claude Code working in
  this repo.
- `.claude/settings.json` — Claude Code permission profile sealed against a
  hash in `.pragma/claude-settings.hash`.

### 2. Add requirements as you learn the code

Describe each piece of behaviour you care about. One invocation per
requirement. Use `--permutation` once per input/output pair you want to
exercise in tests.

```shell
pragma spec add-requirement \
    --id REQ-001 \
    --title "User can log in" \
    --description "Operator signs in with email and password." \
    --touches src/auth/login.py \
    --permutation 'valid|Correct email and strong password returns a JWT|success' \
    --permutation 'bad_password|Wrong password is rejected with 401|reject'
```

Add as many as you need. The IDs must be unique — `pragma spec
add-requirement` refuses to clobber an existing entry. Run `pragma freeze`
after each batch of edits.

### 3. Freeze the manifest

```shell
pragma freeze
```

This reads `pragma.yaml`, validates the schema, and writes
`pragma.lock.json` atomically. The lock holds a SHA-256 over the canonical
form of the manifest — if the YAML drifts even a whitespace, the hook
blocks the commit.

### 4. Sanity check

```shell
pragma verify all
```

`verify all` runs `verify manifest` + `verify gate` + `verify discipline` +
`verify integrity`. All four must return `ok: true`.

### 5. Commit

```shell
git add pragma.yaml pragma.lock.json .pre-commit-config.yaml PRAGMA.md .claude/
git commit -m "chore: adopt pragma"
```

The `pre-commit` stage runs `pragma verify all` before the commit lands;
`commit-msg` runs `pragma verify message` against your draft; `pre-push`
runs `pragma verify commits` across the range about to be pushed. A
malformed commit is rejected before it touches history.

## Greenfield quickstart

Use greenfield mode when you have an empty directory and want Pragma to
scaffold the first manifest from a problem statement.

### 1. Initialise

```shell
mkdir demo && cd demo
pragma init --greenfield --name demo --language python
```

This writes the seed manifest (`pragma.yaml` with `mode: greenfield`, one
milestone `M01`, one slice `M01.S1`, one placeholder `REQ-000`), primes
`.pragma/state.json`, writes `pragma.lock.json`, and drops a `claude.md`
at the repo root so Claude Code knows the house rules.

### 2. Describe the problem

Create `docs/problem.md` with one `#` heading per capability the system
needs:

```markdown
# Register
Users create an account with email and password.

# Login
Users sign in with their email and password.

# Logout
Users sign out and invalidate the session.
```

### 3. Plan the greenfield

```shell
pragma spec plan-greenfield --from docs/problem.md
```

This is deterministic (no LLM). It reads the headings, replaces the seed
`REQ-000` with one `REQ-00N` placeholder per heading inside `M01.S1`, and
leaves a `TODO(pragma)` marker in the description field of each.

### 4. Fill in each TODO

Either open the manifest in an editor:

```shell
$EDITOR pragma.yaml
```

…or let Claude Code do it by calling `pragma spec add-requirement` for
each requirement it wants to flesh out. Every `TODO(pragma)` needs a real
description and at least one `--permutation`.

### 5. Freeze

```shell
pragma freeze
```

### 6. Ship the first slice

```shell
pragma slice activate M01.S1
# write failing tests under tests/ — one per permutation, named
#   test_req_<req_number>_<permutation_id>
pragma unlock
# implement src/ until tests are green
pragma slice complete
git commit -m "feat(m01): ship first slice"
```

## The test-first loop

Pragma enforces test-first development via the gate. Once a slice is
active (`pragma slice activate <id>`), the gate is `LOCKED` — you cannot
edit `src/` until every declared permutation has a failing test under
`tests/`.

```shell
pragma slice activate M01.S1
# gate is now LOCKED
pytest tests/ -x            # every declared permutation must fail (red)
pragma unlock               # refuses unless every permutation is red
# gate is now UNLOCKED — edit src/, make tests green
pytest tests/
pragma slice complete       # refuses unless every test is green
```

Why `pragma unlock` refuses on a green test: the gate enforces the
invariant "you cannot write production code until every permutation has
a failing test." If the test already passes, either the behaviour is
already there (the requirement is mis-scoped) or the test doesn't
actually exercise the permutation (the test is a no-op). Both are bugs
you want to catch before you write `src/`.

Test names matter. The gate joins tests to permutations by name:

```text
test_req_<req_number>_<permutation_id>
```

For `REQ-001` with permutation `valid_credentials`, the test function is
`test_req_001_valid_credentials`. A typo makes the test invisible to the
gate.

## What pre-commit catches

The `.pre-commit-config.yaml` that `pragma init` writes installs four
Pragma hooks:

- `pragma-verify-manifest` — `pragma verify manifest`. Blocks if
  `pragma.yaml` and `pragma.lock.json` disagree.
- `pragma-verify-all` — `pragma verify all`. Runs manifest + gate +
  discipline + integrity. Blocks on any failure.
- `pragma-verify-commit-msg` — `pragma verify message`. Runs at
  `commit-msg` stage against the draft.
- `pragma-verify-commits-pre-push` — `pragma verify commits`. Runs at
  `pre-push` against the range about to be pushed.

### Canonical commit shape

Subject is `≤ 72` characters. Body must contain a `WHY:` paragraph and a
`Co-Authored-By:` trailer. `WHAT:` and `WHERE:` are conventional but not
required. Example:

```text
feat(auth): add email+password login with JWT issuance

WHY: The product brief requires an authenticated landing page. This is
  the minimal slice that unblocks every downstream feature that needs a
  `user_id` in request context.
WHAT: New `src/auth/login.py` with `authenticate(email, password)` that
  returns a signed JWT or raises `BadCredentials`. Wired into the
  existing Flask app under `/login`.
WHERE: src/auth/login.py (new), src/app.py.

Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>
```

### When pre-commit blocks your push

`pragma verify all` writes JSON to stderr. Read the `code` field. Common
blockers:

- `manifest_hash_mismatch` — you edited `pragma.yaml` without running
  `pragma freeze`. Fix: `pragma freeze && git add pragma.lock.json`.
- `gate_locked` — you edited `src/` while the gate was `LOCKED`. Fix:
  `pragma unlock` (after making every permutation red), or revert your
  `src/` edits.
- `discipline_failed` — a test name doesn't match a declared permutation
  (or vice-versa). Fix the test name or add the permutation.
- `commit_shape_invalid` — subject over 72 chars, missing `WHY:`, or
  missing `Co-Authored-By:`. Reword the message.

`pragma doctor` is the catch-all diagnostic for anything else — see
[`doctor.md`](doctor.md).
